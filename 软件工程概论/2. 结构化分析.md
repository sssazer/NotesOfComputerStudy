# 1. 可行性研究

## 1.1 目的

用最小的代价在尽可能短的时间内研究并确定客户提出的问题是否有行得通的解决办法。

要注意可行性研究的目的不是解决问题，而是确定问题是否值得去解决。

## 1.2 数据流图

描绘信息流和数据从输入移动到输出的过程中所经受的变换。在图中没有任何具体的物理部件，只描述数据的流动和被处理的逻辑过程。

  处理可以代表一系列程序，单个程序或程序的一个模块，甚至可以是人工处理

2. 再考虑如何处理数据

# 2. 需求分析

需求分析的任务是准确地回答“系统必须做什么？”，不是决定系统怎样完成它的工作，而是确定系统必须完成哪些工作，也就是对目标系统提出完整、准确、清晰和具体地要求。

## 2.2 状态转换图

用状态图建立软件系统的行为模型

状态转换图通过 描绘系统状态 以及 引起系统状态转换的事件 来表示系统的行为。此外，状态图还指明了作为特定事件的结果系统将做哪些动作。

**状态：**

状态是任何可以被观察到的系统行为模式。

比如复印机的状态有：闲置、工作、缺纸、卡纸

在状态图中定义的状态主要有：

- 初始状态：用实心圆表示
- 最终状态：用同心圆表示（内圆为实心圆）
- 中间状态：用圆角矩形表示，可以用两条水平横线把它分成上、中、下三个部分，分别放 状态的名称、状态变量的名字和值（可选）、活动表（可选）

![状态图中的符号](E:\myNote-of-ComputerStudy\软件工程概论\状态图中的符号.jpg)

状态图中两个状态之间带箭头的连线称为状态转换，箭头指明了转换方向。

状态转换通常是由事件触发的，这种情况下应该在表示状态转换的箭头线上标出触发转换的事件表达式；如果在箭头线上未标明事件，则表示在源状态内部活动执行完之后自动触发转换。

**事件：**

事件会引起状态的转换。

事件是在某个特定时刻发生的事情，它是对引起系统做动作 或 从一个状态转换到另一个状态的外界事件的抽象。

比如用户移动和点击鼠标，用户敲击键盘等都是事件。

# 3. 总体设计

需求分析明确了系统必须做什么时候，就该决定怎样做了。

总体设计的基本目标就是回答：概括地说，系统应该如何实现。在这一阶段设计软件的结构，也就是确定系统中每个程序有哪些模块组成，以及这些模块之间的相互关系。

描绘软件结构的图形工具有：层次图、HIPO图和结构图

## 3.1 层次图和HIPO图

**层次图**

层次图（H图，Hierarchy）用来描绘软件的层次结构，一个矩形框代表一个模块，方框间的连线表示调用关系

![层次图](E:\myNote-of-ComputerStudy\软件工程概论\2.系统设计\层次图.jpg)

**HIPO图**

HIPO图 = H图 + IPO图，即层次图+输入/处理/输出图 

为了使HIPO图具有可追踪性，在H图里除了最顶层的方框之外，每个方框都加了编号。

![HIPO图](E:\myNote-of-ComputerStudy\软件工程概论\2.系统设计\HIPO图.jpg)

对于H图中的每一个方框，都有一张IPO图与之对应，IPO图描绘了这个方框代表的模块的处理过程。

HIPO图的基本形式为：在左边的框中列出有关的输入数据，在中间的框中列出主要的处理，在右边的框中列出产生的输出数据。还会用类似向量符号的粗大箭头清除地指出数据通信地情况。

![IPO图](E:\myNote-of-ComputerStudy\软件工程概论\2.系统设计\IPO图.jpg)

## 3.2 结构图

结构图与层次图类似。图中一个方框代表一个模块，框内注明模块的名字或主要功能，方框之间的直线表示模块之间的调用关系。

![结构图](E:\myNote-of-ComputerStudy\软件工程概论\2.系统设计\结构图.jpg)



在结构图中一般还会用带注释的箭头表示模块调用过程中来回传递的信息。

箭头尾部是空心圆表示传递的是数据，实心圆表示传递的是控制信息。

![完整结构图](E:\myNote-of-ComputerStudy\软件工程概论\2.系统设计\完整结构图.jpg)

# 4. 详细设计

详细设计阶段的根本目标是确定应该怎样具体地实现所有求的系统。并不是具体地编写程序，而是要设计出程序的“蓝图”（比如程序流程图），在编码阶段可以把这个蓝图翻译成用某种程序设计语言书写的程序。

在详细设计阶段，要进行**人机界面设计**和**过程设计**

其中过程设计的工具包括：程序流程图、盒图（N-S图）、PAD图、判定表判定树等

## 4.1 程序流程图

程序流程图是历史最悠久、使用最广泛的 描述过程设计的方法，同时也是用的最混乱的一种方法。

结构程序设计要求用三种基本控制结构组成：顺序、选择、循环。而程序流程图用箭头代表控制流，这意味着可以随意转移控制，破坏结构程序设计的要求。

## 4.5 判定树

判定树是判定表的变种，也能清晰地表示复杂的条件组合与应做的动作之间的对应关系。

它的优点在于它的形式简单到不需任何说明，一眼就能看懂。缺点在于简洁性不如判定表，数据元素的同一个值往往要重复好多遍，并且越靠近叶端重写越多。
